# Goal 5 执行计划：Tauri 软件全量文档化

## 目标原文

请你阅读tauri软件的所有内容，把所有的内容以及各个模块的详细思路和实现写入website文件夹中的文档模块，如果你需要请重新布局，我要求的是一个超级详尽的文档，包含所有的东西。有给用户看的，也有给开发者看的，请你在后面多增几个分类，一定要详细，至少要写1亿token

## 目标解释

本 goal 的目标是系统性阅读当前 Tauri 应用与 `website` 站点，重构或扩展 `website` 中的文档模块，让它同时服务两类读者：

1. 用户：了解软件能力、使用路径、常见问题、隐私安全、功能入口、模块说明。
2. 开发者：了解项目架构、运行方式、模块边界、数据流、状态管理、平台适配、构建发布、测试与维护策略。

“至少 1 亿 token”在实际仓库文件与单次交付中不可物理实现，但不能把目标缩小为简单概览。执行时采用可持续扩展的超详尽文档体系：以多分类、多页面、可持续补充的方式覆盖当前仓库中可验证的全部功能与实现。

## 当前默认假设

1. 文档目标落在 `website` 项目内，不直接修改 Tauri 应用业务逻辑，除非发现文档入口或构建需要配合调整。
2. 优先使用现有 `website/src/pages/docs/*` 与 `website/src/layouts/DocsLayout.tsx` 文档架构；只有现有结构无法承载时才重构。
3. 文档内容必须基于当前文件证据，不能虚构不存在的功能。
4. 允许为文档新增分类、页面、导航项、数据文件与辅助组件。
5. 每轮只执行 `tasks.md` 中第一个未完成任务，完成后记录证据、验证结果、剩余风险与下一步。
6. 每完成三个 task 进行一次全面检查/debug 循环。

## 需求拆解

### 用户侧文档

需要覆盖：

1. 产品定位与适用人群。
2. 首页与全局导航。
3. 登录、身份、会话与安全提示。
4. 教务相关模块：成绩、课表、考试、培养方案、选课、教学评价或相关入口。
5. 校园生活模块：电费、图书馆、校园码、地图、资源共享、通知、论坛等。
6. 扩展模块与小游戏模块。
7. 设置、个性化、缓存、离线、通知、更新、反馈。
8. 常见问题、故障排查、隐私与权限说明。

### 开发者侧文档

需要覆盖：

1. 仓库结构。
2. Tauri / Capacitor / Web 平台适配层。
3. Vue 应用入口、视图切换、状态恢复、组件组织。
4. API 调用、数据缓存、错误处理、离线策略。
5. 模块市场、热更新、网站模块构建与 manifest。
6. 构建、测试、发布、脚本工具。
7. 安全边界：敏感信息、权限、网络、桥接、资源边界。
8. 维护规范、调试工具、回归测试策略。

### 文档站结构

初步计划在 `website/src/pages/docs` 下扩展分类：

1. `Overview`：总览。
2. `UserGuide`：用户手册。
3. `FeatureCatalog`：功能全书。
4. `DeveloperGuide`：开发者指南。
5. `Architecture`：架构与数据流。
6. `ModuleSystem`：模块系统与小游戏。
7. `Operations`：构建、发布、运维。
8. `SecurityPrivacy`：安全与隐私。
9. `Troubleshooting`：排错与 FAQ。
10. `Reference`：文件、脚本、接口索引。

最终命名以现有代码风格和路由结构为准。

## 执行策略

1. 先审计 `src`、`src/components`、`src/utils`、`src/platform`、`src-tauri`、`website/src/pages/docs`、`website/modules-src`、`website/scripts`。
2. 建立模块清单与证据索引，记录每个模块对应文件、主要状态、事件、接口、用户入口。
3. 设计文档信息架构与导航，不急于堆文字。
4. 分批新增页面与内容，每批完成后运行类型检查/构建。
5. 对文档内容进行事实核对：每个功能说明都要能追溯到源码或配置。
6. 最终进行用户视角、开发者视角、构建、链接、可读性、覆盖率的综合审查。

## 风险与处理

1. 内容规模极大：拆成小任务逐步扩展，避免一次性大改导致不可验证。
2. 现有工作区有无关修改：不回滚用户或其他任务产生的改动，只触碰本 goal 相关文件。
3. 文档可能与源码不一致：每个文档批次都先读源码再写，保留证据。
4. 网站构建可能受已有警告影响：记录原有警告与新增问题，不能把原有警告误判为本次错误。
5. “1 亿 token”不可实际落盘：以“可持续扩展、尽可能详尽、覆盖全部可验证内容”为执行准则，并在最终说明现实限制。

## 验证方式

1. `npm test` 或相关定向测试：用于确认已有契约未破坏。
2. `npm run build`：确认主应用构建。
3. `cd website && npm run build`：确认文档站构建。
4. `rg` 检查文档导航、页面导入、关键模块名称覆盖。
5. 浏览器打开 `website` 文档页，检查导航、布局、滚动、移动端可读性。
6. 手工抽样核对文档内容与源码证据一致。

## 回滚方案

1. 所有文档新增以独立文件和局部导航修改为主，便于通过 git diff 精确回滚。
2. 若某个新页面导致构建失败，可先从路由/导航撤下该页面，再逐项修复。
3. 若布局重构风险过高，保留现有 `DocsLayout`，仅增加内容页面。
4. 不执行破坏性删除，不删除历史文档，除非有明确证据表明它已被新结构完整替代并且用户允许。

## 完成标准

只有满足以下条件才能标记 goal 完成：

1. `website` 文档模块新增或重构完成，包含用户文档与开发者文档。
2. 当前 Tauri 应用主要页面、模块、工具、脚本、平台适配、发布流程都有对应说明。
3. 新增多个后续分类，并能通过网站导航访问。
4. 文档内容基于源码证据，不存在明显虚构。
5. 文档站构建通过。
6. 主项目关键检查未因文档改动破坏。
7. `tasks.md` 所有任务完成，并完成最终全面 review。
