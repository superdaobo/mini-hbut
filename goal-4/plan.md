# 论坛前后端重建计划

## 需求分析

目标是在 Mini-HBUT Tauri 应用中新增并完善“论坛模式”，入口位于底部导航栏中心。论坛前端是本地固定页面，但所有动态数据通过后端请求和刷新。整体体验需要接近“虎扑式社区”：发帖、回帖、评分、收藏、关注、举报、通知、个人页、管理页、附件/图床能力、缓存、防重复提交提示，以及可支撑约 2000 DAU 的稳定读写路径。

后端需要基于现有 `D:\Documents\C_learn\成绩查询\ocr-service` 融合论坛能力，并和当前 OCR 服务部署目标保持一致。数据策略为：论坛结构化数据进入 SQLPub 中新建论坛表；附件、图片、备份等对象数据优先进入 HF Private Bucket；为了降低 HF 账号或空间异常导致的数据丢失风险，设计 OneDrive 定时同步备份路径。图床接口优先做后端代理上传和访问，避免依赖不可控免费外部图床。

前端必须严格套用 `D:\Download\stitch_1_1_homepage_replication (4).zip` 的视觉风格，并覆盖论坛每个页面，不允许只做部分页面或风格漂移。需要重点修复上一版“太丑”的问题：先提取 Stitch 样式基线，再用设计 token、布局组件和页面级验收约束重做论坛 UI。

## 风险

- 数据库结构变更、线上 SQLPub 写入、HF Space 推送、HF Secret 修改、读取或写入 `C:\Users\yangd\Desktop\mini-hbut-ocrupload-secrets.txt` 都属于高风险操作，执行前必须获得用户明确确认。
- 免费外部图床在中国大陆可用性、限额、合规和长期稳定性不可控。默认方案应是“自有后端图床代理 + HF Private Bucket + OneDrive 备份”，外部图床只作为调研结果或可选降级，不作为主链路。
- HF Private Bucket、OneDrive 同步、SQLPub 共用数据库涉及密钥与权限边界，不能把真实密钥提交到 Git 或 HF 仓库。
- 论坛用户数据和成绩/排名数据同库，需要严格隔离表名、索引、写入路径和权限，避免影响已有查询业务。
- Tauri 本地页面与后端跨域、离线、弱网、重复提交、缓存失效都需要显式处理，否则用户体验会明显下降。
- Stitch 视觉包需要先解压和审阅，不得凭记忆或主观审美重做。

## 执行方案

1. 第一阶段只做目标初始化和规划，不改代码。
2. 每轮只执行 `tasks.md` 中第一个未完成任务；每个任务完成后更新 `tasks.md`，有代码修改则提交 commit。
3. 先恢复上下文与现状审计：读取 AGENTS、参考视觉包、前端现有论坛实现、后端论坛实现、测试状态、Git 脏改动边界。
4. 后端按接口契约优先：补论坛 API 契约测试，再完善 SQLPub/SQLite store、HF Bucket 对象存储、附件代理、缓存头、用户/管理读模型、备份接口。
5. 前端先提取 Stitch 视觉基线：颜色、字体、间距、卡片、导航、按钮、列表、输入框、反馈状态。再用这些 token 重构所有论坛页面。
6. 前端页面覆盖：广场 Feed、帖子详情、发帖、通知/私信、我的、个人主页、管理页、上传/图床状态、空状态、错误状态、加载状态。
7. 缓存与防重复：后端提供 `Cache-Control`/`ETag`/条件请求；前端提供 TTL 缓存、stale fallback、写操作后精确失效、pending action 防重复提交和 toast 提示。
8. 性能与体验：列表分页、轻量响应、图片代理缓存、移动端优先布局、避免文本溢出和重复点击、弱网可理解错误。
9. 每三个任务后执行一次大型全面检查-debug 循环，覆盖需求偏离、测试、构建、UI、性能、安全、数据一致性、回滚。
10. 最终验收前运行后端全量测试、前端全量 Vitest、前端 build、浏览器移动端/桌面端 UI 验收。HF 推送和线上测试必须先走危险操作确认。

## 验证方式

- 后端：
  - 运行 `ocr-service` 全量 pytest。
  - 覆盖 SQLite 与 SQLPub store 的论坛表、帖子、回复、评分、收藏、关注、举报、附件、备份、缓存响应。
  - 用本地临时目录验证 HF Bucket 缺失时的降级路径和附件代理缓存。
- 前端：
  - 运行 `tauri-app` 全量 Vitest。
  - 运行 `npm run build`。
  - 用浏览器在 390x844、430x932、768x1024、1280x800 验收论坛入口和每个论坛页面。
  - 核查 UI 是否严格贴近 Stitch 包：颜色、圆角、阴影、间距、页面密度、移动端适配。
  - 验证发帖、收藏、回复、评分、上传等重复点击会被阻止并给出提示。
- 集成：
  - 本地前端连本地或测试后端，验证主要 API 请求、缓存、错误状态。
  - HF Space 推送与线上测试只在用户确认后执行。

## 回滚方案

- 每个任务独立 commit，问题可按任务回滚。
- 不删除用户已有改动和构建产物；只在明确确认后处理高风险文件或远程资源。
- 数据库变更优先使用幂等建表/迁移语句，不做 destructive migration。
- HF Bucket 与 OneDrive 备份都采用新增路径和可配置环境变量，不覆盖已有 OCR 数据。
- 前端重构保留论坛入口契约，若新 UI 出现阻断，可回退单个 `ForumView` 和 `forum_api` 相关 commit。

## 默认假设

- 论坛默认优先做“后端图床代理 + HF Private Bucket 存储”，不默认接第三方免费图床。
- SQLPub 密钥只在用户确认高风险操作后读取；若测试阶段没有密钥，则使用 SQLite/local stub 完成可验证开发。
- OneDrive 同步先实现可配置脚本和接口，不在未确认前写入真实账号或远程盘。
- “严格按照 Stitch 样式”意味着先以 zip 内页面为视觉源，不再使用上一版自定义 Campus Vitality 风格。
- 管理能力先做最小可用：举报列表、用户搜索、封禁/解封、徽章发放、备份记录；不引入复杂审核流。
- 所有关键说明、任务记录和新增关键流程注释使用简体中文。
