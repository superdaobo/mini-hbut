# 首页天气温度条与天气图标修复计划

## 需求摘要

修复首页天气页面的两个视觉与逻辑问题：

1. 未来天气里的温度条需要根据每一天的最低气温与最高气温，在整体温度范围内进行正确缩放，不能只显示固定短条或位置失真。
2. 温度条颜色需要根据实际气温决定，最低气温不应固定为蓝色；如果最低气温本身偏热，也可以呈现浅红、橙色等更符合温度感知的颜色。
3. 下雨、阴天、晴天等天气图标当前颜色反差过大，需要降低突兀感，让不同天气状态之间有更自然的过渡效果。

## 上下文

- 项目目录：`d:\Documents\C_learn\成绩查询\tauri-app`
- 当前活动文件：`release_dev.py`
- 目标页面：应用首页天气卡片或天气弹窗中的未来天气列表。
- 用户提供了截图，显示未来天气温度条当前像固定短渐变片段，最低温文本为蓝色、最高温文本为红色，天气图标蓝/灰反差明显。

## 默认假设

1. 天气首页由前端代码渲染，可能是 React/Vue/Svelte/Tauri 前端的一部分。
2. 未来天气数据里已有每日 `min` 与 `max` 温度，或可从现有字段解析出最低/最高气温。
3. 温度条应按所有展示天数的全局最低温与全局最高温归一化，这样不同日期之间可横向比较。
4. 图标颜色修复优先沿用现有图标体系，不引入新的大型依赖。
5. 如果现有测试框架可用，优先补最小单元测试；如果缺少测试框架，则补可运行的纯函数测试或最小验证脚本。

## 需求拆解

### 温度条缩放

- 计算展示列表中的全局最低温 `globalMin` 与全局最高温 `globalMax`。
- 每日温度条区间：
  - 左偏移：`(dayMin - globalMin) / (globalMax - globalMin)`
  - 宽度：`(dayMax - dayMin) / (globalMax - globalMin)`
- 当 `globalMin === globalMax` 或数据无效时，使用稳定兜底，避免除零和布局崩溃。
- 设置最小可见宽度，避免温差过小时完全不可见，但不能破坏相对位置。

### 温度颜色

- 不把低温端固定为蓝色，而是用温度到色阶的映射。
- 建议色阶：
  - 低温：柔和蓝
  - 温和：青绿或绿色
  - 偏热：橙色
  - 高温：柔和红
- 每日温度条起止颜色分别由 `dayMin`、`dayMax` 映射得到。
- 文本颜色也可使用同一温度色阶，但需要保证对比度。

### 天气图标颜色

- 保留天气类型辨识度，但降低极端饱和和明暗反差。
- 建议按天气类别建立语义色：
  - 晴：暖黄/橙，饱和度适中。
  - 多云/阴：中性蓝灰，避免过黑。
  - 雨：柔和蓝，避免与文本蓝强冲突。
  - 雪/雾/其他：使用低饱和冷色或灰蓝。
- 图标可使用渐变或同一色系的主/辅色，让过渡更自然。

## 技术方案

1. 定位天气页面组件、天气数据模型、温度条样式和图标渲染逻辑。
2. 抽出或复用纯函数：
   - `getTemperatureRangeScale(...)`
   - `getTemperatureColor(...)`
   - `getWeatherIconTone(...)`
3. 先写失败测试覆盖：
   - 全局范围缩放正确。
   - 高最低温时低温端不再固定蓝色。
   - 阴/雨/晴图标色调更接近同一柔和视觉系统。
4. 修改组件样式，保证条形宽度、偏移和渐变颜色来自计算结果。
5. 运行测试、类型检查、构建；如果可启动本地应用，则用浏览器验证截图区域。

## 风险

- 现有天气数据字段可能命名不统一，需要先追踪数据流。
- 若图标是第三方图标组件，可能只能通过 `color`/`className` 控制，无法细分主辅色。
- 如果项目缺少测试框架，自动化测试范围会受限，需要采用可执行的最小验证。
- 视觉修复容易出现对比度不足，需要检查浅色背景下的可读性。

## 验证方式

1. 运行新增或相关单元测试，验证温度范围计算和颜色映射。
2. 运行项目现有静态检查、类型检查或构建命令，单次命令设置不超过 60s 超时。
3. 启动本地前端或 Tauri 开发服务后，用浏览器查看首页天气页面。
4. 检查未来天气列表：
   - 温度条区间随每日最低/最高气温变化。
   - 低温端颜色由实际温度决定，不固定蓝色。
   - 雨、阴、晴图标视觉更柔和，反差不过大。

## 回滚方案

- 所有业务变更通过 patch 和 git diff 可追踪。
- 若修复引发构建或视觉问题，可回滚天气组件和新增测试相关文件。
- 若只是一组颜色 token 不合适，可保留缩放逻辑，单独调整颜色映射。
