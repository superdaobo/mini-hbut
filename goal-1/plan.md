# Goal 1 规划：website 游戏模块完善

## 目标原文

请你完成下列任务，首先请你查看website里面的游戏模块，请你优化跳出湖工大，笨鸟先飞的页面，确保能内嵌进tauri里面，并且适应竖屏，然后请你优化跳出湖工大的页面和游戏逻辑，建筑太丑了而且容易穿模，请你修复，并且请你多做几个游戏，比如大富翁，矿工等等，可以借用网上的开源代码并且融合湖工的特色，请你完全规划实施，确保能玩，好玩，完善

## 需求拆解

1. 查看并理解 `website` 里的现有游戏模块结构、构建方式、发布目录和 Tauri 内嵌入口。
2. 优化“跳出湖工大”页面，使其能稳定内嵌进 Tauri，适配竖屏移动端。
3. 优化“笨鸟先飞”页面，使其能稳定内嵌进 Tauri，适配竖屏移动端。
4. 优化“跳出湖工大”的视觉表现，重点修复建筑太丑的问题。
5. 优化“跳出湖工大”的游戏逻辑，重点修复容易穿模的问题。
6. 新增多个湖工特色小游戏，明确包含类似“大富翁”“矿工”等玩法方向。
7. 新增游戏必须能玩、好玩、相对完善，并能进入现有 website 模块体系。
8. 可以参考开源玩法思路，但不能直接引入不清楚许可证或来源的整包代码；优先自行实现轻量核心逻辑，降低版权和集成风险。

## 默认假设

1. “跳出湖工大”和“笨鸟先飞”都是 `website` 模块系统中的独立游戏模块。
2. Tauri 内嵌主要通过现有模块宿主加载 `website/public/modules` 或构建产物实现。
3. 竖屏优先目标视口为手机宽度 360-430px，高度 720-932px，同时兼容桌面预览。
4. 新增游戏优先做轻量、可维护、无额外重依赖的 HTML/CSS/JS 游戏，避免引入复杂构建链。
5. 如果已有构建脚本会同步 `website/modules-src` 到 `website/public/modules`，应以源码目录为主，构建目录为产物。

## 执行策略

1. 先做结构审计，确认模块源码、manifest、catalog、构建脚本、Tauri 宿主加载协议。
2. 先修现有两个游戏的内嵌和竖屏体验，再深入修“跳出湖工大”的物理和视觉。
3. 将游戏核心逻辑拆出可测试函数，优先用 Vitest 或轻量 Node 脚本验证碰撞、视口、manifest、资源路径。
4. 新增游戏按“一个可玩闭环”最小完成：开始、操作、计分、失败/胜利、重开、移动端触控。
5. 每完成三个 task 做一次大型检查-debug循环，避免累计 UI、构建、数据和集成问题。

## 风险

1. `website/public/modules` 可能是构建产物，直接改可能被覆盖。
2. 游戏如果使用 canvas/physics，竖屏比例和高 DPI 下容易出现穿模、输入延迟或碰撞误差。
3. Tauri WebView 对路径、iframe、触摸事件、滚动、音频自动播放可能有限制。
4. 引用网上开源代码存在许可证和集成风险。
5. 新增多个游戏范围较大，需要分阶段交付，不能把“能打开”当作“好玩完善”。

## 验证方式

1. 静态检查：manifest/catalog/schema/资源路径检查脚本或测试。
2. 单元测试：关键游戏逻辑，如碰撞检测、地图移动、得分、失败判定。
3. 构建验证：项目现有构建命令能成功运行。
4. 浏览器验证：本地打开或 dev server 中检查模块列表、游戏启动、竖屏布局、触控输入。
5. Tauri 内嵌验证：至少确认模块宿主加载路径、iframe/bridge 尺寸、滚动和安全区适配。
6. 视觉验证：移动端截图或 Playwright viewport 检查，无横向滚动、主要控件可见、游戏区域不被遮挡。

## 回滚方案

1. 所有改动保持在明确文件范围内，按 task 记录。
2. 每个 task 完成后记录 diff、验证命令和剩余风险。
3. 如果新游戏造成构建或宿主异常，可以先从 catalog/manifest 移除入口，保留源码待修。
4. 不删除用户已有未提交变更；如构建产物冲突，先记录并隔离处理。

## 完成标准

1. “跳出湖工大”和“笨鸟先飞”能在竖屏移动端和 Tauri 宿主中正常显示、操作和重开。
2. “跳出湖工大”的建筑视觉明显改善，碰撞逻辑不再高频穿模。
3. 至少新增多个湖工特色游戏，其中包含大富翁类和矿工类玩法，并接入模块目录。
4. 所有游戏均有基础可玩闭环、移动端输入、计分/状态反馈、重开机制。
5. 相关测试、构建、页面验证通过；大型检查记录无未解决高风险项。
